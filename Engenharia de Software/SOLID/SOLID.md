# SOLID: Os 5 Princípios do Design Orientado a Objetos

Os princípios SOLID são diretrizes fundamentais para criar software que seja fácil de manter, estender e testar. Eles servem como a base para o uso correto de **Design Patterns** e são o coração da **Clean Architecture**.

---

## 🏛️ Os 5 Princípios Detalhados

1.  **S - Single Responsibility Principle**
    *   Foco em ter apenas uma razão para mudar.
    *   [[SRP - Single Responsibility Principle|Detalhes e Exemplos]]
2.  **O - Open/Closed Principle**
    *   Aberto para extensão, fechado para modificação.
    *   [[OCP - Open-Closed Principle|Detalhes e Exemplos]]
3.  **L - Liskov Substitution Principle**
    *   Subclasses devem ser substituíveis por suas classes pai.
    *   [[LSP - Liskov Substitution Principle|Detalhes e Exemplos]]
4.  **I - Interface Segregation Principle**
    *   Interfaces pequenas e específicas são melhores que interfaces grandes.
    *   [[ISP - Interface Segregation Principle|Detalhes e Exemplos]]
5.  **D - Dependency Inversion Principle**
    *   Dependa de abstrações, não de implementações.
    *   [[DIP - Dependency Inversion Principle|Detalhes e Exemplos]]

---

## 🚀 Por que usar SOLID?
*   **Facilidade de Manutenção:** Alterações em uma parte do sistema não causam efeitos cascata inesperados.
*   **Testabilidade:** O desacoplamento permite testar componentes em isolamento usando Mocks.
*   **Escalabilidade:** Novos recursos podem ser adicionados com o mínimo de alteração no código legado.
